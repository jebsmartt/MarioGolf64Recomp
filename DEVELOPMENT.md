## Three-Repo Pipeline

This project depends on two upstream repositories. The dependency chain is strictly one-directional:

```
mariogolf64 (decomp)
    ↓  produces mariogolf64.elf
N64Recomp/mariogolf branch
    ↓  produces recomp_output/
MarioGolf64Recomp (this repo)
    ↓  produces playable port
```

**mariogolf64** is an independent community decompilation project. It is not aware of this port and should not be treated as a dependency in the traditional sense — it is an input source. Contributions to mariogolf64 should follow that project's own contribution guidelines without reference to this port.

**N64Recomp** (`mariogolf` branch of [jebsmartt/N64Recomp](https://github.com/jebsmartt/N64Recomp)) contains Mario Golf-specific ELF patches and recompiler configuration on top of the upstream N64Recomp tool. See [MARIOGOLF_PIPELINE.md](https://github.com/jebsmartt/N64Recomp/blob/mariogolf/MARIOGOLF_PIPELINE.md) for full technical documentation of that stage.

### Why not ship recomp output in this repo?

The recompiled C files are a mechanical transformation of Nintendo's copyrighted binary. Shipping them would be legally equivalent to distributing a modified version of the original ROM. Users must generate the recomp output themselves from a ROM they own. This is the same legal basis that decompilation and port projects operate on generally.

## How N64Recomp Works

N64Recomp takes an ELF file and statically recompiles each function into a C function. For example, the MIPS instruction:

```asm
addiu $r4, $r4, 0x20
```

becomes:

```c
ctx->r4 = ADD32(ctx->r4, 0x20);
```

Every recompiled function has the signature:

```c
void func_XXXXXXXX(uint8_t* rdram, recomp_context* ctx);
```

where `rdram` is the emulated N64 RAM and `ctx` holds the MIPS register state. Function calls (`jal`) become direct C function calls. Indirect calls (`jalr`) become `LOOKUP_FUNC(ctx->rXX)(rdram, ctx)` — a runtime dispatch that the port must implement.

The recomp output is not standalone — it requires the macro definitions and runtime support provided by librecomp and ultramodern to compile and execute.

## Runtime Architecture

```
recomp_output/          Recompiled game code
      ↓
  librecomp             Bridges recompiled code ↔ ultramodern
                        Also handles: overlay loading, PI DMA (ROM reads),
                        EEPROM/SRAM/Flash saving
      ↓
  ultramodern           Reimplements libultra:
                        Threads, controllers, audio, message queues,
                        timers, RSP task handling, VI timing
      ↓
platform callbacks      Implemented in src/ — game-specific:
                        controller input, audio sample playback
      ↓
    RT64                Graphics renderer (Vulkan)
```

### ultramodern

ultramodern reimplements the Nintendo 64 operating system primitives that the game relies on. Mario Golf uses NuSystem (Nintendo's higher-level framework built on top of libultra), which maps well onto ultramodern's thread and message queue model.

ultramodern expects the integrating project to provide platform callbacks for:
- Reading controller state
- Playing back audio samples
- Registering a graphics renderer

These callbacks are implemented in `src/`.

### librecomp

librecomp bridges the gap between the raw recompiled C and ultramodern. It provides:
- The macro implementations (`ADD32`, `LOOKUP_FUNC`, `RELOC_HI16`, etc.) that the recompiled C uses
- Overlay loading and management
- PI DMA (ROM reads — how the game loads assets from the cartridge)
- Save data handling (EEPROM, SRAM, Flash)

librecomp also handles the `recomp_overlays.inl` file generated by N64Recomp, which describes the game's overlay sections and their runtime addresses.

## Graphics: RT64

RT64 is the recommended renderer for N64Recomp-based ports. It implements the N64's RDP (Reality Display Processor) on modern hardware using Vulkan.

On Linux, RT64 is built with:
- `RT64_STATIC=ON` — static library, self-contained binary
- `RT64_SDL_WINDOW_VULKAN=ON` — SDL2 window with Vulkan surface (required for Linux)

RT64 is registered with ultramodern as the graphics renderer via a callback during initialization.

### Why RT64 over other renderers?

RT64 is the only renderer with proven compatibility with N64ModernRuntime. It is used by Zelda64Recomp in production and supports hardware-accelerated ray tracing for enhanced visuals where available — one of the key improvements over original hardware that this project aims to demonstrate.

## Key Design Decisions

### Why clang over GCC?

N64ModernRuntime was developed and tested with clang. Clang 18 (available on Ubuntu 24.04) exceeds the minimum requirement of clang 15. While GCC 13 should work, using the same compiler as the upstream project reduces the risk of hitting C++20 edge cases. GCC remains a supported fallback.

### Why Linux only (initially)?

The build infrastructure was established on Linux first for simplicity. The `task_win32.cpp` file in ultramodern is fully guarded with `#ifdef _WIN32` and compiles to nothing on Linux — no Linux-specific workaround is needed. Windows and macOS support can be added later by providing the appropriate platform callback implementations and verifying the CMake configuration.

### Why not fork Zelda64Recomp?

Zelda 64 is a much more complete decompilation than Mario Golf. Zelda64Recomp's runtime is deeply entangled with Zelda-specific game systems (scheduler, audio sequencer, actor model, etc.). Stripping those out while simultaneously adapting to Mario Golf's different architecture would add significant complexity with little benefit. Starting from N64ModernRuntime directly produces a cleaner, game-appropriate integration.

Zelda64Recomp remains a valuable reference for how specific systems were solved.

### Why is recomp output not one file per function?

N64Recomp defaults to 50 functions per output file (`functions_per_output_file = 50`). This is a deliberate performance tradeoff — one file per function produces thousands of individual `.c` files which significantly slows compilation due to file I/O overhead. 50 per file produces 8 files which compile quickly while remaining manageable.

## Mario Golf Specific Notes

### Game structure

Mario Golf uses NuSystem, Nintendo's higher-level framework built on libultra. NuSystem provides:
- A main game loop (`nuGfxTaskAllEndWait`)
- Simplified controller management
- Audio management
- Pre-built thread setup

This is simpler than Zelda 64's custom scheduler and should map reasonably well onto ultramodern's abstractions.

### The ADD30 overlay

Mario Golf has a large data overlay section (`.ADD30`, `0x801F4A30`+, ~23MB) that is loaded into RAM at runtime. This section contains game assets rather than executable code — it has no `X` (executable) flag in the ELF. It is not recompiled. Calls from `.main` into ADD30 are handled at runtime via `LOOKUP_FUNC` which librecomp implements.

Functions in `.main` that call ADD30 symbols directly are stubbed in `mariogolf.toml`. See [MARIOGOLF_PIPELINE.md](https://github.com/jebsmartt/N64Recomp/blob/mariogolf/MARIOGOLF_PIPELINE.md) for the full stub list and rationale.

### Save data

Mario Golf uses SRAM for save data. librecomp provides SRAM support out of the box.

### Audio

Mario Golf uses the Nintendo 64's audio list (AL) system via libultra. The AL functions (`alInit`, `alEnvmixerPull`, etc.) are present as section-bound symbols in the recomp output after the ABS symbol patch. ultramodern handles RSP audio task execution; platform-specific audio sample playback must be implemented in `src/`.

## Known Limitations

### Unhandled instructions

N64Recomp does not implement the `tge` (Trap if Greater or Equal) or `cache` instructions. Functions containing these are either stubbed or, in the case of cache functions, present as section-bound symbols whose ABS callers are stubbed.

- `tge` is used as a compiler-generated assertion trap. Stubbed functions lose this behavior, which is non-critical for runtime functionality.
- `cache` instructions perform CPU cache management. On the host platform, cache coherency is handled by the OS and hardware — these operations are safely no-ops.

### func_800A2F50

This function requires both a stub (to skip recompilation) and a binary patch (to prevent static function generation during section scanning). The underlying issue is that the function has a `jr ra` at offset +4 making it appear 0x10 bytes long, while the actual body extends to `0x800A3854` and contains a jump table whose size the recompiler cannot determine. See `patch_elf.py` and `MARIOGOLF_PIPELINE.md` for full details.

### ADD30 overlay functions not recompiled

Approximately 23MB of ADD30 overlay content is not recompiled. The runtime behavior of overlay loading and the `LOOKUP_FUNC` dispatch for ADD30 functions needs to be validated once the port reaches a running state.

### Platform support

Currently Linux (x86_64) only. Windows and macOS support is not yet implemented.

## Next Steps

The project currently builds a stub executable that links all dependencies. The following work is needed to reach a running state:

1. **Platform callbacks** — implement controller input, audio playback, and renderer registration in `src/main.cpp`
2. **ROM loading** — implement PI DMA / ROM read callbacks so the game can load assets

5. **RT64 integration** — register RT64 as the renderer and create an SDL2 window
6. **First boot** — get the game to reach its initialization code without crashing
7. **Graphics output** — verify the RDP command lists are being processed by RT64
8. **Audio output** — verify audio tasks are executing and samples are playing back
9. **Controller input** — verify NuSystem controller polling maps onto ultramodern input
10. **Save data** — verify SRAM reads/writes work correctly via librecomp

# Session Notes (2026-03-18)
## What was completed
### RSP microcode (aspMain)
Mario Golf uses Nintendo's audio DSP (aspMain) for RSP audio tasks. The ucode was located in the ROM at offset 0x0008F250 (text size 0x193C, IMEM text address 0x04001080). RSPRecomp required three fixes to handle Mario Golf's ucode:

- Added DPC_START, DPC_END, DPC_CURRENT to the mfc0 handler (previously only DPC_STATUS was handled)
- Added DPC_START, DPC_END, DPC_CURRENT, DPC_STATUS to the mtc0 handler
- Added cfc2 handler (reads VCO/VCC/VCE control registers — returns 0 for audio ucode)

The ucode contains jumps to low IMEM addresses (0x0014, 0x0108, etc.) that correspond to rspboot — a separately-loaded RSP initialization stub. These are declared as extra_indirect_branch_targets in rsp/aspMain.toml and emit UnhandledJumpTarget stubs in the generated C. This is acceptable since rspboot execution is handled by ultramodern's RSP task infrastructure, not the recompiled ucode path.
Mario Golf does not use njpgdspMain (JPEG decoder). The get_rsp_microcode callback only handles M_AUDTASK.
RSPRecomp fixes are committed to the mariogolf branch of N64Recomp. The generated rsp/aspMain.cpp is in MarioGolf64Recomp/rsp/.
